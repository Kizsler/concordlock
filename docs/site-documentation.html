01

Overview

The Concord Locksmith website is a small, fast, content-driven marketing site for a family-owned locksmith business in downtown Concord, CA, established 1947. The site introduces the business, lists services across six specialties, and accepts customer contact submissions that route to the shop's email.

What the site does

  • Tells the story of a 75+ year family-owned locksmith
  • Markets six service categories: residential, commercial, automotive, safes & vaults, keyless entry, self-defense products
  • Surfaces credentials — licensed, bonded, BBB accredited, Nextdoor Faves 2024
  • Provides phone CTA throughout (the primary conversion path is a phone call)
  • Accepts contact form submissions that email service@concordlock.com
  • Shows business hours, address, and an embedded Google Map

What the site does not do

  • No e-commerce, no online payments, no scheduling/booking
  • No user accounts, login, or authentication
  • No CMS — content is edited directly in HTML files
  • No database — the site is functionally static plus a single email-sending endpoint

Audience

Local customers in Concord and the surrounding Contra Costa County area searching for a trusted locksmith. The site is optimized for trust signals (years in business, credentials, real shop photos) over feature breadth.

System Architecture

The site is a small Node.js/Express application that serves static HTML pages plus a single API endpoint for contact-form email delivery. The whole thing runs in a Docker container on fly.io. Cloudflare manages DNS but does not proxy traffic, so requests reach fly.io directly after the DNS lookup.

Domain & DNS

Two services share responsibility for the domain. Network Solutions is the registrar — the company you pay annually to keep the name. Cloudflare is the DNS host — where the actual records (A, AAAA, CNAME, MX, TXT) live and get edited.

How they connect

  1. The domain concordlock.com is registered at Network Solutions.
  2. At Network Solutions, the nameservers for the domain have been changed to martha.ns.cloudflare.com and nile.ns.cloudflare.com. This delegates DNS resolution to Cloudflare.
  3. All DNS records are managed in the Cloudflare dashboard at dash.cloudflare.com.
  4. For the website records, Cloudflare is in "DNS only" mode (gray-cloud icon next to each record). Cloudflare resolves the lookup but doesn't proxy traffic — the user's browser connects to fly.io's IP directly.

Where to make a DNS change

  • For website / domain records: log in to Cloudflare → Websites → concordlock.com → DNS → Records.
  • To change registrar info, billing, or transfer the domain: log in to Network Solutions.
  • To change the nameservers themselves (e.g., move DNS away from Cloudflare): that happens at Network Solutions under "Manage" → nameserver settings. Don't change this unless you're consciously moving DNS.

Important notes

  • Domain registration is separate from web hosting. Cancelling the old WordPress hosting does not affect the domain. Keep the registration paid at Network Solutions; let the WordPress hosting expire.
  • If you move email providers (Google Workspace, Microsoft 365, etc.), update the MX records in Cloudflare, not Network Solutions. Cloudflare is now the source of truth for records.
  • The Resend domain verification records live in Cloudflare. If Resend ever shows the domain as "unverified," check those TXT/CNAME records are still present.
05

SSL / HTTPS

HTTPS is provided by fly.io, which automatically issues and renews Let's Encrypt certificates as long as the DNS records point at fly. There is nothing to configure manually beyond running fly certs add once per hostname.

Current certificates

  • concordlock.com — Let's Encrypt, auto-renewed
  • www.concordlock.com — Let's Encrypt, auto-renewed
  • concordlock.fly.dev — managed by fly.io's wildcard cert (always works)

How fly.io issues certs

When you run fly certs add yourdomain.com, fly.io initiates a Let's Encrypt ACME challenge. Because the DNS already points at fly's IPs, fly answers the HTTP-01 challenge automatically. The cert is issued within seconds and renews itself every ~60 days.

HTTPS enforcement

Plain HTTP requests are 301-redirected to HTTPS by fly's edge (force_https = true in fly.toml). Browsers will only ever see the HTTPS version.

If HTTPS breaks

Run fly certs check concordlock.com -a concordlock to see cert status. If issuance is stuck, the most common cause is DNS pointing somewhere other than fly.io. Verify with nslookup concordlock.com — it should return 66.241.124.67.

Contact Form & Resend

The contact form on /contact sends a real email to service@concordlock.com via Resend. The submission is JS-driven (the form does not reload the page).

End-to-end flow

  1. User fills out the form (Name, Email, Phone, Service, Message).
  2. JavaScript intercepts the submit, packs the values into JSON, POSTs to /api/contact.
  3. Express server validates the input (required fields, email format, length limits).
  4. Server calls Resend's API with the authenticated API key.
  5. Resend sends the email from noreply@concordlock.com to service@concordlock.com, with the customer's address as the Reply-To.
  6. Server returns {"ok": true}.
  7. JS shows a success message and resets the form.

Configuration (fly secrets)

SecretDescription
RESEND_API_KEYThe Resend API key. Sensitive — never committed.
RESEND_TORecipient address (service@concordlock.com).
RESEND_FROMDisplay + sender address (Concord Locksmith Contact <noreply@concordlock.com>).

Email receiving

Resend handles sending. Receiving mail at service@concordlock.com depends on the MX records in DNS pointing to a mailbox provider (Google Workspace, Microsoft 365, or similar). If submissions stop landing, check both Resend's "Emails" dashboard (was it sent? was it delivered?) and the mail provider's spam folder.

Resend domain verification

The domain concordlock.com is verified in Resend, which means emails are sent with proper DKIM/SPF signatures and are far less likely to be marked as spam. If the domain ever fails re-verification, those records can be re-checked at resend.com → Domains.

Quick check: did a submission go through?

1. Resend dashboard → Emails tab → recent sends should show "delivered."
2. Check the inbox at service@concordlock.com (and spam).
3. fly logs -a concordlock — the server logs Resend errors if the API call failed.

Deployment Workflow

The workflow is intentionally simple: edit files locally, commit to git, push to GitHub for backup, then run fly deploy. There is no CI/CD pipeline (none is needed for this scale).

The full deploy cycle

# 1. Make your changes in the working tree
# 2. Commit them to git
git add .
git commit -m "Describe what changed and why"

# 3. Push to GitHub (backup; doesn't deploy)
git push origin main

# 4. Deploy to fly.io
fly deploy

What fly deploy does, step by step

  1. Reads fly.toml to know which app to deploy.
  2. Tarballs the local working directory (excluding entries in .dockerignore).
  3. Uploads to fly's remote builder (Depot).
  4. Builds a Docker image from Dockerfile (downloads node:20-alpine, runs npm install, copies files).
  5. Pushes the image to registry.fly.io.
  6. Performs a rolling update on the two machines: stops one, replaces it, verifies health, then does the other. Zero downtime if everything goes well.
  7. Verifies DNS is correct and the app is reachable.

Rolling back a deploy

If a deploy goes wrong, the previous image is still in the registry. To roll back:

# Show recent deployments
fly releases -a concordlock

# Roll back to a specific version (the column labeled VERSION in releases output)
fly deploy --image registry.fly.io/concordlock:deployment-<ID>

Local development

# From inside the project folder:
npm install --omit=dev

# Set the env vars locally and run:
RESEND_API_KEY=re_... RESEND_TO=service@concordlock.com node server.js

# Site is then at http://localhost:8080

Note: a quick static-only preview can also be done with python -m http.server 8080, but the contact form won't work without the Node server.

Gotchas & Known Quirks

1. Cold-start delay on the first visit

Because auto_stop_machines is enabled, the fly.io machines stop after a few minutes of no traffic. The first request after idle takes ~1–2 seconds to wake up a machine. Subsequent requests are normal speed. This is expected behavior, not a bug. If you want zero cold starts, set min_machines_running = 1 in fly.toml — but that costs more.

2. Browser cache holds old assets

After you deploy a change, your own browser may show the old version because it cached main.js or style.css. A hard refresh (Ctrl+Shift+R on Windows, Cmd+Shift+R on Mac) bypasses the cache. Customers without dev tools will get the new version naturally as their cache expires.

3. The hero video on the homepage

The hero video is hotlinked from Pexels' CDN, not stored locally. If Pexels ever changes their URL structure or removes the clip, the hero will break visually. If that happens, swap to a different clip in index.html (the <source> tag) and in main.js (the playlist array).

4. The contact form requires the Node server

Static-file-only previews (e.g., opening contact.html directly in a browser, or python -m http.server) will show the form but submissions will fail because there's no /api/contact handler. Always test contact form changes with the actual Node server running.

5. URL casing matters on the server but is normalized in browsers

/Services is technically a different URL than /services on the server. Browsers usually preserve the case from links, so as long as all internal links use lowercase, this isn't an issue. Just don't manually type capitalized URLs into the address bar.

6. The Google Maps iframe uses a baked-in lat/long

The embedded Google Maps URL on /index and /contact renders by latitude/longitude, with "1804 Colfax Street, Concord, CA" as the search hint. If the displayed address ever changes, update both the visible address and the search hint inside the iframe URL.

7. Direct .html URLs redirect

If you share an old link like concordlock.com/services.html, it 301-redirects to concordlock.com/services. This is intentional and good for SEO; just don't be surprised by the redirect.

Cloudflare's role (DNS only, today)

Cloudflare is in your stack as the DNS host. It is not in the request path for actual HTTPS traffic to the site — that goes directly from the user to fly.io after the DNS lookup.

What Cloudflare is doing right now

  • Authoritative DNS for concordlock.com on the free plan. Anyone resolving the domain gets their answer from Cloudflare's nameservers.
  • Hosting all DNS records — web (A/AAAA/CNAME), email (MX, TXT/SPF, DKIM, DMARC), and any others.
  • A nicer dashboard than Network Solutions for editing records, with faster propagation.

What Cloudflare is NOT doing right now

  • Not proxying or caching site content (records are "DNS only" / gray cloud).
  • Not providing DDoS protection at the edge for HTTPS traffic.
  • Not running a Web Application Firewall in front of the site.
  • Not collecting traffic analytics for the site.

How to enable Cloudflare's proxy / CDN if you want to

Switching from "DNS only" to "Proxied" (gray cloud → orange cloud) routes traffic through Cloudflare's network. To do this safely:

  1. In Cloudflare, go to SSL/TLS → set encryption mode to Full (Strict). This makes Cloudflare verify fly.io's Let's Encrypt cert on the back end.
  2. In Cloudflare's DNS panel, click each web record (the A, AAAA on apex; the CNAME on www) and toggle the proxy status to Proxied (orange cloud).
  3. Wait a few minutes. The site will now resolve to Cloudflare's IPs (e.g., 104.21.x.x) instead of fly.io's IPs. Cloudflare terminates SSL and forwards to fly.io.
  4. Test: curl -I https://concordlock.com should now show cf-ray and server: cloudflare in the headers.

Should you enable proxy mode?

For a small marketing site like this, the gain is modest. fly.io already has anycast edge, Let's Encrypt is already auto-renewed, traffic is low. Enabling Cloudflare's proxy adds:

  • Edge caching of static files (CSS, JS, images)
  • DDoS protection
  • Bot/spam mitigation if the contact form ever attracts abuse
  • Traffic analytics in the Cloudflare dashboard

It's a low-cost thing to flip on later if you ever notice slow loads, bot traffic, or want analytics. There's no urgency today.
